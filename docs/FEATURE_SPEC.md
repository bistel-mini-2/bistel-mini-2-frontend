## 0. 바이브코딩용 사용법

이 문서는 사람이 읽는 기능명세서이면서, AI에게 기능 구현을 맡길 때 기준 문서로 쓰는 것을 전제로 한다.

바이브코딩할 때는 아래 원칙을 먼저 AI에게 전달한다.

```text
이 문서를 기준으로 구현한다.
기능명, 분류, 담당, 우선순위를 임의로 바꾸지 않는다.
공통 기능은 각 화면에 중복 구현하지 말고 재사용 모듈로 만든다.
추천, 지원 가능성 분석, 비교, 신청, 챗봇은 공통 모듈을 호출하는 방식으로 연결한다.
상태가 "시작 전"인 기능은 구현 전 TODO 또는 이슈 단위로 분리한다.
통합 비고에 적힌 병합 기준은 기능 분리보다 우선한다.
```

AI에게 한 번에 전체 기능을 만들게 하기보다, 아래 단위로 나눠 요청하는 것이 좋다.

| 구현 단위 | 우선순위 | 완료 기준 |
| --- | --- | --- |
| 회원/인증 기반 | 상 | 회원가입, 로그인/로그아웃, 인증/인가가 동작한다. |
| 사용자 조건 프로필 | 상 | 가족 프로필을 입력/저장하고 추천/분석에서 재사용한다. |
| 정책 탐색 기본 | 상 | 정책 목록, 검색, 상세 화면이 동작한다. |
| 추천 기본 흐름 | 상 | AI 조건 분석, 후보 검색, 추천 결과 카드가 연결된다. |
| RAG 검색/근거 | 상 | 정책 chunk 검색 결과와 근거를 공통 모듈로 반환한다. |
| 지원 가능성 분석 | 상 | `check_eligibility` 결과와 근거를 표시한다. |
| 비교 기능 | 상/중 | 비교 바구니, 비교 정보 수집, 비교 결과를 연결한다. |
| 신청 준비 | 상/중 | 신청 정보, 서류, 체크리스트, 주의사항을 연결한다. |
| 챗봇 | 상/중 | 채팅, 의도 분류, 흐름 연동, 스트리밍, 안전 안내를 연결한다. |
| 마이페이지 | 중/하 | 프로필, 관심정책, 체크리스트, 추천/상담 이력을 모아 보여준다. |

구현 프롬프트 예시는 다음처럼 쓴다.

```text
docs/feature-specification.md를 기준으로 정책 탐색 기능을 구현해줘.
범위는 "정책탐색" 분류만이다.
우선순위 상 기능인 정책 목록, 정책 상세, 정책 검색을 먼저 구현해줘.
정책 선택 컴포넌트와 정책 설명 생성 엔진은 공통 모듈로 재사용 가능하게 연결해줘.
```

바이브코딩 시 특히 흔한 실수는 다음과 같다.

- 공통 기능을 각 화면에 중복 구현하는 것
- 추천 조건 입력과 분석 조건 입력을 별도 모델로 만드는 것
- 정책 AI 요약과 챗봇 RAG 답변을 서로 다른 엔진으로 만드는 것
- 비교/분석/신청의 정책 선택 UI를 각각 따로 만드는 것
- 챗봇에서 추천/자격/비교/신청 로직을 다시 구현하는 것
- 우선순위 `하` 기능을 먼저 구현해서 핵심 흐름이 늦어지는 것
- `통합 비고`의 통합 기준을 무시하고 과거 기능번호대로 쪼개는 것

## 2. 구현 우선순위

### 2.1 1차 구현 대상

우선순위 `상` 기능은 서비스 핵심 흐름을 구성하므로 먼저 구현한다.

| 기능명 | 분류 | 담당 | 설명 |
| --- | --- | --- | --- |
| 로그인/로그아웃 | 회원/마이페이지 | 추천/회원 | 로그인·로그아웃 처리 |
| 사용자 조건(가족 프로필) 입력·저장 | 공통 | 추천/회원 | 가족 상황 입력 후 프로필로 저장. 추천·분석이 재사용 |
| 추천 결과 | 추천 | 추천/회원 | TOP N 추천 카드 + 추천 이유(공통 RAG 사용) |
| 인증/인가 | 회원/마이페이지 | 추천/회원 | 토큰 기반 인증, 권한 처리 |
| 정책 후보 검색 | 추천 | 추천/회원 | 조건으로 DB 후보 검색 |
| AI 조건 분석 | 추천 | 추천/회원 | 사용자 입력을 조건 태그로 변환 |
| 신청 체크리스트 | 신청 | 추천/회원 | 생성 + 체크상태 관리 통합 |
| 회원가입 | 회원/마이페이지 | 추천/회원 | 이메일 기반 회원가입 |
| 정책 설명 생성 엔진 | 공통 | 탐색/상세 | 정책 문서를 쉬운 설명으로 변환 |
| 정책 목록 | 정책탐색 | 탐색/상세 | 전체 정책 조회 |
| 정책 상세 | 정책탐색 | 탐색/상세 | 기본정보 + AI요약 + 다음 액션 |
| 정책 검색 | 정책탐색 | 탐색/상세 | 키워드 검색 |
| 신청 정보 조회 | 신청 | 탐색/상세 | 신청방법/기간/문의처 조회 |
| 비교 정보 수집 | 비교 | 판단/비교 | 2개 정책 정보·RAG 수집 |
| RAG 검색/근거 | 공통 | 판단/비교 | 정책 chunk 의미 검색 + 근거 반환 |
| 지원 가능성 판단·결과 | 지원가능성분석 | 판단/비교 | 충족/미충족/추가확인 분류 + 결과 표시 |

| 스트리밍 (SSE) | 챗봇 | 챗봇/신청 | 답변 토큰 SSE 스트리밍 |
| 안전 안내 | 챗봇 | 챗봇/신청 | 단정 금지 + 공식기관 확인 면책 문구 |

## 4. 공통 모듈 기준

아래 기능은 여러 화면에서 재사용해야 하므로 중복 구현하지 않는다.

| 공통 모듈 | 재사용 위치 | 기준 |
| --- | --- | --- |
| 사용자 조건 프로필 | 추천, 지원 가능성 분석, 비교, 신청, 챗봇 | 조건 입력과 분석 조건 입력을 하나의 프로필/조건 모델로 통합한다. |
| 정책 선택 컴포넌트 | 분석 대상 선택, 비교 대상 선택, 신청 정책 선택 | 리스트/검색 기반 선택 UI를 공통화하고, 비교는 다중 선택 옵션으로 확장한다. |
| 정책 설명 생성 엔진 | 정책 상세, 챗봇 답변 | 정책 문서를 쉬운 설명으로 바꾸는 로직은 하나만 두고 화면별 렌더링만 분리한다. |
| RAG 검색/근거 | 추천, 지원 가능성 분석, 비교, 정책 상세, 챗봇 | `search_policy_chunks`를 공용 검색/근거 반환 모듈로 둔다. |
| 신청 체크리스트 | 신청 준비, 마이페이지 | 생성과 체크상태 관리를 통합하고 마이페이지에서 진행 상태를 조회한다. |
| 이력 조회 | 추천 이력, 상담 이력, 마이페이지 | 저장은 각 흐름에서 수행하고 조회는 마이페이지에서 모아 보여준다. |

## 5. 담당별 작업 묶음

### 5.1 추천/회원

담당 범위:

- 회원가입
- 로그인/로그아웃
- 인증/인가
- 관심 정책
- 마이페이지
- 사용자 조건(가족 프로필) 입력·저장
- 추천 후보 검색
- AI 조건 분석
- 추천 결과
- 다시 추천
- 추천 이력
- 신청 체크리스트

핵심 방향:

추천/회원 파트는 사용자 계정, 프로필, 추천 흐름, 마이페이지의 기반을 담당한다. 특히 가족 프로필은 추천과 분석에서 함께 쓰이므로 별도 추천 전용 모델로 만들지 않는다.

### 5.2 탐색/상세

담당 범위:

- 정책 목록
- 정책 검색
- 정책 필터/정렬
- 정책 상세
- 관련 정책
- 정책 설명 생성 엔진
- 정책 선택 컴포넌트
- 신청 정보 조회
- 서류 확인
- 주의사항

핵심 방향:

탐색/상세 파트는 정책 데이터 조회와 상세 화면의 중심이다. 정책 설명 생성 엔진과 정책 선택 컴포넌트는 다른 흐름에서도 재사용되므로 공통 모듈로 구현한다.

### 5.3 판단/비교

담당 범위:

- RAG 검색/근거
- 지원 가능성 판단·결과
- 비교 정보 수집
- 비교 바구니
- 비교 결과

핵심 방향:

판단/비교 파트는 근거 기반 판단과 비교 흐름을 담당한다. RAG 검색/근거는 모든 AI 흐름에서 쓰이는 공통 모듈이므로 단일 소유자를 정하고 인터페이스를 안정적으로 유지한다.

### 5.4 챗봇/신청

담당 범위:

- 채팅
- 스트리밍 (SSE)
- 의도 분류/라우팅
- 흐름 연동(호출)
- 대화 맥락(슬롯)
- 안전 안내
- 상담 이력

핵심 방향:

챗봇은 추천/자격/비교/신청 로직을 내부에 새로 만들지 않는다. 의도 분류 후 기존 함수 또는 Graph를 호출하고, 결과를 대화체로 변환한다.

## 6. 구현 체크리스트

기능 구현 전 확인:

- 기능명과 분류가 문서와 일치하는가?
- 우선순위 `상` 기능부터 구현하고 있는가?
- 공통 기능을 화면별로 중복 구현하지 않았는가?
- `통합 비고`에 있는 병합 기준을 반영했는가?
- 저장과 조회 책임이 분리된 기능은 저장 위치와 조회 위치가 명확한가?

기능 구현 후 확인:

- 담당 그룹별 핵심 흐름이 끊기지 않는가?
- 추천/분석/비교/신청/챗봇에서 RAG 근거 반환 방식이 일관적인가?
- 챗봇이 기존 흐름을 호출하고 있는가?
- 마이페이지에서 프로필, 관심 정책, 체크리스트, 이력을 모아 볼 수 있는가?
- 안전 안내와 면책 문구가 정책 판단/챗봇 응답에 반영되는가?
